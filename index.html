Introdução

A World Wide Web conhecida como Web 1.0 é caracterizada de um lado, por um pequeno grupo de usuários especialistas responsáveis por desenvolver os documentos (HTML e XHTML) a serem disponibilizados na Web. Do outro lado, tem-se um grande número de usuários que apenas navega nesse ambiente, não sendo possível a eles publicar qualquer informação.

No intuito de possibilitar que os usuários não especialistas também criem e compartilhem informações na Web, surgiu a Web 2.0, também conhecida como a “Web de Documentos”. A Web 2.0 é caracterizada pelas comunidades de usuários (e.g. Facebook) e ferramentas de colaboração (e.g. Wikis) [Cormode & Krishnamurthy, 2008].

O crescimento contínuo e desordenado de documentos na Web 2.0 tem dificultado a localização de conteúdos, mesmo com o auxílio dos atuais mecanismos de busca (e.g. Google), o que em geral demanda muito esforço de um usuário humano na análise e cruzamento dos resultados. Neste cenário, pode-se imaginar, por exemplo, um turista que deseja organizar uma viagem, usando o conteúdo disponível na Web, para assistir aos jogos da copa no Brasil (Figura 1). Para realizar essa tarefa, o turista terá que consultar diversos sites diferentes como: empresas aéreas, para compra de passagens; hotéis, para estadia; informações sobre jogos da copa em páginas de esporte; e transporte, para locomoção. O que acontece em geral é esses dados serem analisados por um usuário humano, e não por aplicações computacionais, pelo fato dos documentos serem publicados na Web em diferentes formatos (e.g. textual, vídeo), diferentes idiomas (e.g. português, inglês), apresentarem estruturas diferentes para os resultados (e.g. listas, tabelas), dentre outros.

Figura 1 – Web 2.0 aplicada à organização de uma viagem para assistir aos jogos da copa no Brasil.

Nesse contexto, a Web Semântica, também conhecida como “Web de Dados”, surge como uma iniciativa para estruturar e conectar os dados de forma que as aplicações computacionais possam processar os resultados e realizar o cruzamento de dados de forma automática, simplificando, no caso da Figura 2, a tarefa do usuário de organizar a viagem para assistir aos jogos da copa no Brasil.

Figura 2 – Web Semântica aplicada à organização de uma viagem para assistir aos jogos da copa no Brasil.

       Como parte do desenvolvimento da Web Semântica tem-se o conceito de Linked Data (Dados Conectados ou Dados Ligados) que tem a proposta de conectar todos os dados disponibilizados na Web, pertencentes a diferentes fontes, para formar um único espaço de dados global [W3C, 2014].

Linked Data

O Linked Data é caracterizado por um conjunto de melhores práticas para publicar e conectar os dados estruturados na Web usando um formato que seja compreensível não apenas por usuários humanos, mas também por aplicações computacionais [Bizer et al., 2006].

       Essas melhores práticas baseiam-se em quatro princípios definidos por Berners-Lee (2006) como recomendações para publicação dos dados na Web:

  • Usar URIs para identificar “coisas” (dados)
  • Usar HTTP URIs para que pessoas possam encontrar essas “coisas” (dados)
  • Prover informação útil usando padrões (RDF e SPARQL)
  • Incluir links para outros URIs possibilitando a descoberta de outras “coisas” (dados)

O primeiro princípio refere-se ao uso de URI (Uniform Resource Identifier) que estabelece um padrão de codificação universal para representação e identificação dos dados na Web. Vale ressaltar que cada dado presente na Web deve possuir uma identificação (URI) única.

O segundo princípio baseia-se na utilização do protocolo HTTP como mecanismo de acesso universal para que os dados possam ser localizados na Web a partir de sua URI.

O terceiro princípio está relacionado às informações oferecidas pelos dados para que aplicações computacionais possam compreendê-los e analisá-los a partir dos padrões RDF e SPARQL . O padrão RDF é usado para descrição dos dados na Web e o padrão SPARQL para consulta aos dados.

O quarto princípio define a necessidade de conectar ilimitadamente os dados que estão na web, permitindo que todo tipo de informação disponível na Web possa ser encontrada.

Web de Dados

A proposta da Linked Data é transformar a Web em um enorme banco de dados global, com o intuito de criar uma “Web de Dados”, dando significado às informações armazenadas nos documentos para que aplicações computacionais também possam processar essas informações.

Um paralelo mostrando as diferenças e similaridade entre Web de Documentos (Web 2.0) e a Web de Dados é apresentado na Tabela 1.

Considerando a característica Links, a Web de Documentos emprega os hiperlinks para navegar entre as páginas. Essa estrutura não fornece nenhuma informação que diferencie um documento do outro, inviabilizando que aplicações computacionais possam fazer essa distinção. Por outro lado, na Web de Dados, os links RDF são usados para acessar os dados localizados em diferentes fontes. Diferente dos hiperlinks essa estrutura possibilta às aplicações computacionais extrair informações dos dados.

Em relação aos Mecanismos de Identificação e Acesso, ambos utilizam URIs e o protocolo HTTP, respectivamente.

O Formato para representação dos conteúdos usado na Web de Documentos é baseado na linguagem HTML (Hipertext Markup Language), enquanto a Web de Dados utiliza o padrão RDF.

       Finalmente, a Web de Dados utiliza como linguagem de consulta o padrão SPARQL para recuperar às informações disponibilizadas na Web.

O Projeto Linking Open Data

       Quando se fala em Linked Data um importante aspecto a ser considerado é que nem sempre os dados se encontram em formato aberto. Um dado em formato aberto define dados com livre acesso, que podem ser utilizados, reutilizados e compartilhados por qualquer pessoa. No intuito de incentivar a publicação desse tipo de dados, a comunidade W3C criou em 2007 o projeto Linking Open Data (LOD). Esse projeto incentiva diferentes comunidades a disponibilizarem seus dados em formato aberto na Web, de forma estruturada por meio da tecnologia RDF e seus formatos RDF/XML, N-Triple, Turtle ou N3 , para descrever e conectar os dados de diferentes fontes.

       O projeto LOD herda os princípios da Linked Data, também conhecidos como “Sistema de 5 Estrelas” (Figura 4), que classifica por meio de estrelas o grau de abertura dos dados.

Figura 4 – Sistema de 5 Estrelas. Fonte: Berners-Lee (2006).

Um dado é pontuado com uma estrela se for disponibilizado na Web, em qualquer formato, mas obrigatoriamente com licença aberta.

Para obter duas estrelas, o dado deve atender ao requisito anterior e estar em um formato estruturado, cujo processamento possa ser realizado por uma aplicação computacional. Por exemplo, ao invés de publicar uma imagem escaneada de uma tabela na Web, recomenda-se disponibilizar essa informação utilizando um arquivo no formato excel.

Para adquirir três estrelas, o dado deve estar na Web em um formato estruturado e não proprietário, ou seja, ao invés do formato excel utilizar o formato CSV.

Um dado será classificado como quatro estrelas se estiver na Web em um formato estruturado e não proprietário, utilizando os padrões estabelecidos pela W3C (RDF e SPARQL).

Finalmente, um dado será cinco estrelas se atender a todas as exigências anteriores, com o acréscimo de vincular seus dados aos dados de outras pessoas.

Exemplo de Aplicação

Como exemplo de aplicação dos princípios do Linked Data tem-se o projeto Open Government. Esse projeto foi criado nos Estados Unidos e adotado no Brasil como “Governo Aberto”, o qual apresenta uma ampla divulgação dos dados governamentais aos cidadãos e organizações, tendo como objetivo contribuir para a transparência desses dados e a sua reutilização. A publicação desses dados busca atender a questões relacionadas ao LOD como interoperabilidade, integração dos dados em larga escala, distribuição e acesso livre( O Globo, 2011).

Outro exemplo de aplicação está relacionado a empresa BBC que publica grande quantidade de conteúdos na Web como texto, áudio e vídeo. Essa empresa tem integrado seus dados usando a DBPedia, que é a comunidade responsável por extrair todos os dados da Wikipedia e disponibilizá-los em formato RDF na Web para que pessoas e/ou aplicações possam ser conectar a eles. Cabral et al. (2012) também utiliza os dados da DBPedia como um experimento para publicar os dados do Exame Nacional do Ensino Médio (ENEM) do ano de 2008.

Mais exemplos de aplicações que utilizam os princípios do Linked Data podem ser encontrados em W3C Linked Data Applications .

Padrões RDF e SPARQL

Considerando a pilha de linguagens e tecnologias oferecidas pela Web Semântica (Figura 3) o Linked Data engloba duas tecnologias: RDF e SPARQL. O RDF é utilizado para modelar os dados usando uma estrutura padrão para que esses dados sejam publicados na Web. Essse formato estruturado necessita de um modo de armazenamento específico constituído pelas Triple Stores . Já o SPARQL é uma linguagem de consulta usada para o consumo dos dados nesse ambiente (Figura 8).


Figura 8 – Utilização da Triple Store na visão das tecnologias RDF e SPARQL

Não é o objetivo desse tutorial qualificar o leitor especificamente em RDF ou SPARQL, no entanto, a compreensão das duas tecnologias é relevante para o entendimento do conceito geral de Linked Data.

RDF

O RDF (Resource Description Framework) é um framework ue provê uma estrutura padrão para representar informações na Web, permitindo que os dados sejam associados, apresentados e compartilhados entre diferentes aplicações computacionais. O RDF utiliza uma estrutura de links URIs para nomear o relacionamento entre os dados na Web, o qual pode ser representada em forma de grafo.

A essência da sintaxe RDF é um conjunto de triplas constituídas por sujeito-predicado-objeto, sendo o sujeito e o objeto representados pelos nós no grafo, e o predicado representado pela aresta, conforme ilustrado pela Figura 9.

Figura 9 - Representação gráfica de um Tripla RDF.

Pode-se imaginar um sujeito como sendo uma entidade do tipo "Cidade" que é descrita por uma propriedade "name" cujo valor é "São Paulo" (Figura 10). Logo, sujeito, predicado e objeto estão relacionados aos conceitos de entidade, propriedade e valor.

Figura 10 - Exemplo de entidade, propriedade e valor.

Além de "name" uma entidade pode ser descrita por outras propriedades diferentes. No caso da Figura 11, foram adicionadas as propriedades “population” e “area” a entidade “Cidade” cujos valores são, respectivamente, “43.663.669” e "1530".

Figura 11 - Exemplo de entidade com múltiplas propriedades.

Na Web, sujeitos e predicados são obrigatoriamente representados por URIs. Logo, tudo na Web que queira referenciar esse sujeito ou predicado deve utilizar sua URI. Objetos podem ou não possuir uma URI. Quando ele possui uma URI e possui predicados e valores, esse objeto é considerado um sujeito para seus predicados.

A Figura 12 representa o sujeito e os predicados, apresentados na Figura 11, utilizando suas respectivas URIs. A expressão "a:" presente em cada uma das propriedades representa um prefixo para resumir uma URI. Como exemplo, o prefixo "a:" pode singificar "http://ex.com/property". Logo, a propriedade "a:name" poderia ter sido descrita também como "http://ex.com/property/name" no grafo da Figura 12.

Figura 12 - URIs identificando a entidade e as propriedades em um grafo RDF.

Os grafos RDFs são representações para dados em RDF. Nesse tutorial a linguagem utilizada para descrever os dados é o RDF/XML. O código abaixo apresenta a descrição para o sujeito “Cidade (city)”, com seus respectivos predicados e valores.


<rdf:Description rdf:about="http://ex.com/thing/city">
    <a:name>São Paulo>/a:name>
    <a:population>43663669>/a:population>
    <a:area>1530>/a:area>
</rdf:Description>

A URI que identifica o sujeito é empregada dentro da tag <rdf:Description> para que seja possível reconhecer qual dado está sendo descrito. Cada predicado é representado também por uma tag. No exemplo, o predicado a:name é descrito pela tag de abertura <a:name> e de fechamento <\a:name>, e o objeto ("São Paulo") para esse predicado é colocado entre as tags. A representação dos predicados a:population é a:area, seguem o mesmo raciocínio apresentado para o predicado a:name.

A Figura 13 apresenta um grafo RDF em que duas entidades estão associadas. As URIs identificam de modo único os dois sujeitos. Vale ressaltar que a entidade “Cidade (city)”, nesse exemplo, apesar de ser representado por uma URI (http://ex.com/thing/city) emprega o papel de objeto para o sujeito "Aeroporto (airport)".

Figura 13 - Grafo RDF representando duas entidades associadas.

O código RDF descrevendo a Figura 13 é mostrado na sequência. Cada sujeito possui as tags <rdf:Description> <\rdf:Description> e dentro de cada uma ficam seus respectivos predicados e objetos. Note que na representação do valor do predicado a:location a tag é inicializada com <a:location> e o valor (nesse caso uma URI) é colocado entre aspas em rdf:resource e finalizado apenas com />.


<rdf:Description rdf:about="http://ex.com/thing/airport">
    <a:name>Congonhas</a:name>
    <a:location rdf:resource="http://ex.com/thing/city"/>
</rdf:Description>

<rdf:Description rdf:about="http://ex.com/thing/city">
</rdf:Description>

SPARQL

O padrão SPARQL (SPARQL Protocol and RDF Query Language) proposto pela pela W3C é uma linguagem de consulta projetada para consultar dados armazenados em Triple Store na Web, o qual permite combinar dados provenientes de diferentes fontes.

SPARQL é uma linguagem amplamente usada para buscar dados em formato RDF e é suportada pela maioria das Triple Stores. Essa linguagem é similar a linguagem de consulta SQL (Structured Query Language), suportando agregação, subqueries, negação e criação de valores por meio de expressões. Comparado ao SQL os resultados de consultas SPARQL são dados por conjuntos de grafos RDF e não por tabelas. A estrutura de uma consulta básica SPARQL apresenta as mesmas cláusulas da linguagem SQL, porém com outras funções.


SELECT [DISTINCT] FROM (opcional) WHERE (opcional)

No entanto, o protocolo SPARQL também apresenta funções específicas da linguagem, tais como:

  • FILTER: aplica um filtro ao resultado da consulta
  • LIMIT: limita a quantidade de resultados da consulta
  • PREFIX: cria um “apelido” para uma URI

    • Não serão descritos nesse tutorial conceitos aprofundados sobre SPARQL. No entanto, é necessário entender a estrutura de uma consulta básica e sua relação com uma tripla RDF. Espera-se do leitor um conhecimento intermediários na linguagem SQL para compreender os parágrafos a seguir.

    A Tabela 2 faz um paralelo mostrando o significado das cláusulas SELECT, FROM, WHERE para consultas no formato SQL e SPARQL.


    Tabela 2 – Comparativo entre as cláusulas usadas nas consultas SQL e SPARQL.

    A cláusula SELECT aplicada a uma consulta SQL especifica as colunas de uma ou mais tabelas desejadas no resultado, enquanto que ao ser utilizada em uma consulta SPARQL é especificado o sujeito, predicado e/ou objeto. A cláusula FROM, no lugar de uma tabela, busca os dados em uma ou mais Triple Stores. A cláusula WHERE determina as condições que devem ser atendidas pelo resultado de uma consulta SQL. Já em uma consulta SPARQL as instruções que são colocadas dentro do WHERE restringem o número de resultados.

    Como exemplos de consultas SQL e SPARQL para selecionar todas as cidades de um determinado domínio tem-se:

  • Consulta SQL:
    • USE BDdominio SELECT * FROM Cidade

  • Consulta SPARQL:
    • SELECT ?sujeito FROM <Triple Store> WHERE {
        ?sujeito rdf:type dbpedia-owl:City .
    }


    Retomando a problemática da Web 2.0 apresentada na introdução sobre a dificuldade para analisar os resultados de uma consulta retornados por um mecanismo de busca, imagine que um usuário deseje realizar a seguinte pesquisa na Web: "Recuperar o código postal das cidades dos Estados Unidos que tenham a substring 'ville' no nome". Conforme mostrado pela Figura 16, ao realizar essa consulta usando o mecanismo de busca Google foram retornados 22.700.000 resultados. A tarefa de análise e cruzamento de todas as informações nesse caso teria que ser realizada por um usuário humano.

    Figura 16 - Resultado de uma consulta realizada no Google.

    Considere um novo cenário em que os dados se encontram armazenados na Triple Store do DBpedia (denominada Virtuoso). A DBpedia é uma das comunidades integrantes do projeto LOD que extraiu os dados da Wikipedia e os publicou em formato RDF aberto.

    Vale ressaltar que a sequência de consultas SPARQL para este exemplo foi fracionada em quatro passos, os quais podem ser executados na Triple Store Virtuoso .


  • 1º Encontrar todas as cidades

    • Mais especificamente a consulta SPARQL, apresentada a seguir, deve primeiro "recuperar qualquer sujeito que seja do tipo cidade". Para isso, a cláusula SELECT deve declarar a variável (?Sujeito) que será preenchida com o resultado da busca, e colocar a restrição na cláusula WHERE para que seja aplicada sobre a tripla RDF.


    SELECT ?Sujeito
    WHERE {
        ?Sujeito rdf:type dbpedia-owl:City .
    }
    LIMIT 30

    Na tripla retratada dentro do WHERE, rdf:type descreve a propriedade e dbpedia-owl:City o valor. O RDF Schema disponibiliza mecanismos para descrever grupos de recursos relacionados, esses recursos são utilizados para determinar as caracteríticas de outros recursos, como domínio e alcance. Recursos podem se dividir em grupos chamados classes, e os membros de uma classe são chamados instâncias dessas classes. O vocabulário RDF é definido em um namespace identificado pela URI: . Por conveniência, essa especificação é representada por uma abreviação, o prefixo rdf:. O rdf:type é uma propriedade usada para declarar que um recurso é uma instância de uma classe.

    No código utilizado como exemplo, rdf:type declara que a variável (?Sujeito) deve ser uma instância da classe City. Desse modo é possível restringir o tipo do sujeito que deve ser retornado.

    Do mesmo modo o prefixo dbpedia-owl: é usado para resumir a URI , que representa uma ontologia. Uma ontologia é usada para definir formalmente um domínio, sendo uma alternativa para a integração dos dados. Embora as ontologias sejam usadas como um importante recurso na Linked Data este tutorial não tem o objetivo de descrevê-las, mais informações podem ser encotradas em Breitman (2010).

    Finalmente o ponto ao final da tripla indica o fim da restrição e é obrigatório. Já a cláusula LIMIT é opcional mas é considerada uma boa prática, pois limita o número de triplas retornadas pela consulta. Neste exemplo, a consulta está limitada a exibir as 30 primeiras triplas.


  • 2º Encontrar todas as cidades que estão nos Estados Unidos

    • O resultado da primeira consulta devolve as URIs de todas as entidades que tenham a propriedade type, cujo valor é City. O resultado dessa consulta pode ser visualizado na Figura 17.

    Figura 17 - Resultado da primeira consulta "Encontrar todas as cidades". Fonte: Triple Store Virtuoso.

    Ao clicar na URI http://dbpedia.org/resource/Abbyville,_Kansas da Figura 17, por exemplo, é possível visualizar todas as informações da cidade Abbyville (Kansas) (Figura 18).

    Figura 18 - Informações sobre a cidade Abbyville (Kansas). Fonte: Triple Store Virtuoso

    Para que a consulta retorne todas a cidades que estão nos Estados Unidos é preciso restringi-la a partir da propriedade dbpedia-owl:country com o valor United_States, apresentada à direita na Figura 18.

    A consulta SPARQL, para esse caso, é mostrada a seguir:


    SELECT ?Sujeito
    WHERE {
        ?Sujeito rdf:type dbpedia-owl:City .
        ?Sujeito dbpedia-owl:country dbpedia:United_States .
    }

    Como as duas instruções dentro da cláusula WHERE possuem a mesma variável de retorno (?Sujeito), a consulta acima pode ser reescrita, substituindo o ponto final da primeira instrução por ponto e vírgula e omitindo a variável ?Sujeito da segunda instrução, conforme mostrado a seguir:


    SELECT ?Sujeito
    WHERE {
        ?Sujeito rdf:type dbpedia-owl:City ;
                    dbpedia-owl:country dbpedia:United_States .
    }

  • 3º Recuperar apenas as cidades que possuem "ville" no nome. Por exemplo: Centerville

    • Conforme mostrado na consulta SPARQL, o FILTER, nesse caso, utiliza a função regex para aplicar um filtro no sujeito, retornando apenas os que possuem a substring "ville" em seu nome. O parâmetro "i", é opcional, e quando empregado na função informa ao filtro que deve-se adotar a comparação case-insensitive.

    Vale destacar que o FILTER possibilita o uso de outros funções como bound, isURI, isBlank, dentre outros.


    SELECT ?Sujeito
    WHERE {
        ?Sujeito rdf:type dbpedia-owl:City .
        ?Sujeito dbpedia-owl:country dbpedia:United_States .
        FILTER regex(?Sujeito, “ville", "i")
    }

  • 4º Recuperar o código postal (CEP)

    • O resultado da consulta, nesse caso, deve mostrar, além da URI que identifica a cidade (na variável ?Sujeito), o código postal da cidade . Assim é necessário declarar mais uma variável (?Valor) e incluí-la na cláusula SELECT. Além disso, é necessário adicionar uma nova instrução que especifique a propriedade como dbpedia-owl:postalCode.


    SELECT ?Sujeito ?Valor
    WHERE {
        ?Sujeito rdf:type dbpedia-owl:City .
        ?Sujeito dbpedia-owl:country dbpedia:United_States .
        FILTER regex(?Sujeito, “ville", "i")
        ?Sujeito dbpedia-owl:postalCode ?Valor .
    }

    Exercício 2

    Para praticar os conceitos apresentados sobre SPARQL elabore uma consulta SPARQL que "Selecione todos os jogadores de futebol que jogam ou já tenham jogado no Santos e que nasceram na cidade de Mogi das Cruzes, SP, Brasil. Além disso, a consulta deve retornar a posição do jogador em campo (e.g. lateral, atacante)." Utilize a Triple Store Virtuoso .

    Para facilitar a resolução do exercício, o mesmo foi separado nas seguintes etapas:

    1. Selecione todos os jogadores de futebol
    2. São brasileiros
    3. Nasceram na cidade de Mogi das Cruzes
    4. Tenham jogado no Santos_FC
    5. Mostre a posição em que jogam (e.g. lateral, atacante).

    Solução para o Execício 2

    SELECT ?Sujeito ?Valor
    WHERE {
        ?Sujeito rdf:type dbpedia-owl:SoccerPlayer .
        ?Sujeito dbpedia-owl:birthPlace dbpedia:Brazil .
        ?Sujeito dbpedia-owl:birthPlace dbpedia:Mogi_das_Cruzes .
        ?Sujeito dbpedia-owl:team dbpedia:Santos_FC .
        ?Sujeito dbpedia-owl:position ?Valor .
    }

    Exercício 3

    Como último exercício de fixação, elabore uma consulta SPARQL que retorne as pessoas armazenadas na Triple Store Virtuoso que possuam a substring "angel" no nome. Exiba apenas os 30 primeiros resultados.

    Solução para o Exercício 3

    SELECT ?Sujeito ?Valor
    WHERE {
        ?sujeito rdf:type dbpedia-owl:Person .
        ?sujeito foaf:name ?valor .
        FILTER regex(?valor, "angel", "i")
    }
    LIMIT 30

    Vale ressaltar que apesar dos exemplos acima terem usado apenas a Triple Store do DBPedia é possível realizar consultas SPARQL usando diferentes fontes, conforme apresentado em SPARQL (2013).
